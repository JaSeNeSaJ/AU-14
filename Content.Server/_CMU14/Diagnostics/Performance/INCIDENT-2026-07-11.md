# Server performance incident: 2026-07-11

## Conclusion

The supplied log confirms that a server-wide performance incident was underway by **21:25:12 UTC**, but it does not contain enough telemetry to identify a retained-memory leak or its owner.

The nearest notable pre-incident activity is a transfer of grid `684730` into FTL at 21:23:44, its return at 21:24:31, and the clustered onset report about 40 seconds later. However, the same grid completed an earlier FTL cycle at 21:02:47–21:04:39 without an immediate recorded collapse, which materially weakens the correlation. It remains a reproduction lead, not proof of causation. The log contains no process working set, managed heap size, GC collection/pause data, allocation rate, thread-pool state, entity/component totals, or server FPS/TPS samples at the onset.

## Leads, not conclusions

1. **FTL/grid transfer or PVS work.** The nearest cycle's timing is close and transfer cycles continue throughout the degraded period, but an earlier cycle did not produce an immediate recorded collapse. A reproduction should compare profiler time/allocation for map, transform, PVS, game-state, and Z-level systems before and after a transfer.
2. **ECS retention.** A gameplay system may be spawning entities/components faster than they are deleted. The old log has no counts or prototype/component deltas.
3. **Allocation churn followed by GC pressure.** This can collapse FPS without retaining all allocated memory. The old log has no profiler allocation frames or GC counters at onset.
4. **A non-ECS managed/static cache, worker queue, or native allocation.** Content-side entity tracking cannot prove or exclude these. Process/runtime metrics or two memory dumps are required.
5. **Repeated invalid-coordinate exceptions as an amplifier.** Each error produces a full stack trace. It should be fixed separately, but its presence more than an hour before 21:25 makes it an unlikely initiating event.

## What was downweighted

- The invalid-coordinate `BarricadeSystem.StartDigging` error predates the incident by more than an hour.
- Sporadic lag comments exist earlier (including 20:58 and 21:03 in this round), but the clustered inability-to-move and later server-side/UI reports begin around the reported 21:25 onset.
- Status endpoint traffic occurs both before and after onset.
- Later explosion bursts and stale targeted-projectile/PVS errors occur after users already report lag.
- `serverperf` and profiler commands were run 10–16 minutes after onset.

## Evidence needed next time

- real frame duration, achieved TPS, average FPS, backlog, round state, player/channel count;
- incident opening, reason changes, sustained updates, recovery, worst values, and cooldown suppression;
- rolling entity and component net growth/churn, top prototype/component/map creators, and dirty-entity rate;
- total network send/receive rate and top decoded inbound message types;
- recent complete profiler frames, entity-system and engine time, main-thread allocations, and profiler GC counters;
- low-cardinality metrics plus a wall-clock heartbeat suitable for an external hard-hang alert.

Process RSS/private bytes, retained managed heap, thread-pool queues, and a permanent deadlock still require the engine metrics endpoint and an external collector. See [Server performance diagnostics](README.md).
